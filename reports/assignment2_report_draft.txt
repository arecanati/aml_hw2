1. Introduction

1.1 Problem Definition
This project addresses a binary classification task: predicting whether a client will subscribe to a term deposit based on demographic and campaign-related features. The target variable is encoded as 1 (subscription) and 0 (no subscription).

The objective is to train predictive models that estimate subscription likelihood and classify clients accordingly. Beyond predictive performance, the assignment compares two distinct modeling paradigms: Gradient Boosted Decision Trees (GBDT) and Multi-Layer Perceptrons (MLP), which learn non-linear relationships differently and show different bias-variance behavior.

1.2 Dataset Description
The analysis uses the UCI Bank Marketing dataset, which contains client-level records from direct marketing campaigns.

The dataset includes:
- Numerical features (e.g., age, balance, campaign frequency)
- Categorical features (e.g., job, marital status, education, prior campaign outcome)
- Mixed data quality characteristics including categorical "unknown" values

The target is imbalanced, so model evaluation emphasizes metrics beyond accuracy, specifically precision, recall, F1-score, and AUC-PR.

2. Methods

2.1 Dataset Preparation and Preprocessing

2.1.1 Data Cleaning and Validation
Column names were standardized to lowercase with underscores for consistency. The target variable was converted from categorical labels ("yes"/"no") to binary values (1/0), with explicit validation checks to ensure no unexpected target labels remained.

A data-quality audit was run before splitting, including shape, dtypes, missing-value counts/percentages, "unknown" category counts, and target distribution.

2.1.2 Handling Missing and "Unknown" Values
Numerical missing values were handled via median imputation; categorical missing values used most-frequent imputation.

"Unknown" categorical entries were intentionally kept as explicit categories (not converted to NaN). This preserves potentially useful signal when "unknown" reflects meaningful non-response or unavailable information.

2.1.3 Feature Engineering
Three interpretable engineered features were added:
- prev_contacted: binary indicator of whether the client had been previously contacted (pdays != -1)
- recent_contact: recency bucket derived from pdays (0=never, 1=recent <=30 days, 2=stale >30 days)
- prior_success: binary indicator that previous campaign outcome was success

Feature engineering was intentionally lightweight to preserve interpretability.

2.1.4 Train-Validation-Test Split and Leakage Prevention
A stratified 70/15/15 split was used to preserve class ratios. The same split was used for both model families to maintain fair comparison.

Leakage prevention protocol:
- Split first
- Fit preprocessors only on training data
- Apply fitted transformations to validation/test only via transform

2.2 Model Pipelines

2.2.1 Gradient Boosted Decision Trees (XGBoost)
Model: xgboost.XGBClassifier with binary logistic objective.

Preprocessing:
- Numerical: median imputation
- Categorical: most-frequent imputation + one-hot encoding
- No scaling (tree models do not require feature standardization)

Training:
- Used eval_set with train + validation
- Used early_stopping_rounds to reduce overfitting
- Recorded per-trial training times

2.2.2 Multi-Layer Perceptron (MLP)
Model: sklearn.neural_network.MLPClassifier with Adam optimizer and early stopping.

Preprocessing:
- Numerical: median imputation + StandardScaler
- Categorical: most-frequent imputation + one-hot encoding

Scaling was required for stable neural-network optimization because MLP is sensitive to feature magnitude.

2.3 Hyperparameter Exploration
Primary tuning metric: validation AUC-PR (chosen for class-imbalanced classification).

XGBoost search included:
- learning_rate
- n_estimators
- max_depth
- subsample
- reg_alpha
- reg_lambda

MLP search included:
- hidden_layer_sizes
- activation
- learning_rate_init
- max_iter

A reproducible, computationally budgeted search strategy was used (seeded trial sampling over configured grids).

2.4 Evaluation Protocol
Models were evaluated on validation and test sets using:
- Accuracy
- Precision
- Recall
- F1-score
- AUC-PR

Additional diagnostics:
- XGBoost training vs validation loss
- XGBoost feature importance
- Controlled XGBoost learning-rate comparison (0.01, 0.1, 0.3)
- MLP loss curve (loss_curve_)
- MLP architecture sensitivity
- Test-set precision-recall curves
- Final side-by-side metric and training-time comparison

4. GBDT vs MLP Comparison

4.1 Performance Tradeoff
XGBoost delivered slightly better overall ranking quality (AUC-PR) and slightly better F1, while MLP achieved slightly higher point-estimate accuracy/precision at the default threshold.

4.2 Training Efficiency
XGBoost trained substantially faster than MLP in the best-trial comparison:
- XGBoost best-trial training time: ~0.34 seconds
- MLP best-trial training time: ~26.06 seconds

This runtime gap is substantial for iterative tuning and deployment contexts with tight latency budgets.

4.3 Interpretability
XGBoost provides direct feature-importance outputs, making model behavior easier to inspect. MLP is less transparent and requires indirect diagnostics (loss dynamics, architecture sensitivity, ablations).

4.4 Handling Categorical and Missing Values
Both models relied on the same preprocessing framework (imputation + one-hot for categoricals), ensuring fair treatment. MLP additionally required feature scaling; XGBoost did not.

4.5 Hyperparameter Sensitivity
Both models were sensitive to tuning choices, but not equally across all hyperparameters. XGBoost showed its strongest response to learning rate (with 0.01 clearly underperforming 0.1/0.3), moderate sensitivity to depth and subsampling, and comparatively weaker shifts from regularization terms in this search budget. MLP showed strong sensitivity to learning-rate initialization (0.1 was consistently worse), plus moderate sensitivity to architecture (depth/width), while activation and max-iteration effects were smaller.

4.6 When to Prefer GBDT vs MLP
Prefer GBDT (XGBoost) when:
- The data is tabular with mixed feature types and limited feature engineering budget
- You need strong AUC-PR/F1 performance with relatively fast training/tuning cycles
- You need clearer model diagnostics via feature importance and more straightforward error analysis

Prefer MLP when:
- You are willing to invest in heavier preprocessing/tuning and longer training time
- You expect benefits from richer representation learning (for example, with larger datasets or more complex feature interactions)
- You prioritize a single end-to-end differentiable model pipeline and can tolerate lower interpretability

5. Discussion

5.1 Key Findings
- On this tabular, mixed-type dataset, XGBoost provided the best overall ranking-oriented performance (AUC-PR).
- MLP remained competitive on accuracy and precision, had slightly lower AUC-PR overall, and required much longer training.
- Lightweight engineered interaction-history features improved modeling context while preserving interpretability.

5.2 Bias-Variance Reflection
- XGBoost controls variance via shrinkage, subsampling, and regularization with early stopping, yielding stable generalization.
- MLP has higher optimization variance and stronger sensitivity to scaling and optimization hyperparameters.
- Validation-to-test degradation for both models was moderate, indicating reasonable generalization rather than severe overfitting.

5.3 Limitations
- Hyperparameter search was budgeted rather than exhaustive.
- Evaluation used a single stratified split rather than full cross-validated model selection.
- Threshold tuning for business-specific precision-recall operating points was not optimized beyond default 0.5.

5.4 Future Improvements
- Add cross-validated tuning for more robust model selection.
- Perform probability calibration and decision-threshold optimization.
- Expand feature interactions and domain-specific temporal/contact-history features.
